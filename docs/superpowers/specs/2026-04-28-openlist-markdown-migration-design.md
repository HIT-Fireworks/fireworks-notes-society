# OpenList 与 Markdown 课程树全面迁移设计

## 背景

当前站点使用 VitePress 构建，页面和侧边栏由仓库中的手写 Markdown 目录决定。资料下载区域通过 `<OList>` 组件访问 OpenList 的 `/Fireworks` 目录。实际维护中，OpenList 文件目录比站点 Markdown 课程树更活跃，导致站点入口、课程页和 `<OList path>` 逐渐与远端文件目录失配。

已确认的核心问题不是单个页面路径写错，而是维护模型没有明确区分两类权威源：站点内部结构必须以手写 Markdown 为准，OpenList 只能作为资料目录参考和校验对象。此次“全面迁移”指维护者或 Agent 手动调整 Markdown 课程树，使站点入口尽量覆盖并匹配 OpenList 当前目录结构，同时通过只读审计工具持续发现漂移。

## 目标

1. 明确手写 Markdown 是站点路由、侧边栏、页面标题、课程说明和页面存在性的唯一权威源。
2. 以 OpenList `/Fireworks` 目录作为外部资料目录参考，持续发现本地 Markdown 与远端资料目录之间的差距。
3. 建立只读审计工具，校验 `<OList path>` 是否存在，并报告远端有而本地无、本地有而远端无、重命名候选和孤儿页候选。
4. 手动迁移现有课程树，优先修复数学学院失效路径，补齐缺失顶层分类和高价值课程页。
5. 改善 `OListItem.vue` 的运行时错误处理，避免 OpenList 路径失效时破坏页面正文阅读体验。
6. 消除普通 VitePress 构建中的 OpenList 网络硬依赖，包括现有 data loader 的构建期远端拉取。
7. 更新维护文档，让贡献者知道新增或移动课程页时必须校验 OpenList 绑定路径。

## 非目标

1. 不允许脚本自动创建、移动、删除或修改课程 Markdown 页面。
2. 不把 OpenList 目录作为 VitePress 构建时的直接生成源。
3. 不处理旧 URL 迁移、重定向或兼容路径。
4. 不要求普通 `bun run docs:build` 依赖远端 OpenList 网络可用性。
5. 不在本轮重构 OpenList 后端、下载线路或校园网检测业务逻辑。
6. 不一次性要求所有远端目录都必须补齐课程说明内容；可以先建立入口页和报告，再逐步补正文。

### 顶层分类覆盖不足

OpenList 根目录当前包含 `【公共课】`、`电信学院`、`化工与化学学院`、`环境学院`、`机电工程学院`、`交通科学与工程学院`、`经管学院`、`人文社科学部`、`生命科学和医学学部`、`实验报告`、`数学学院`、`土木工程学院`、`未来技术学院`、`物理学院`、`薪火笔记社-竞赛`、`薪火笔记社-PPT及报告模板`、`询问中`、`仪器学院`。

本地可通过 sidebar 进入的顶层课程分类只有 `【公共课】`、`交通科学与工程学院`、`人文社科学部`、`数学学院`。大量远端资料只能从根目录 `<OList />` 浏览，缺少站内搜索、侧边栏和课程说明入口。

当前本地缺失的顶层入口包括：`电信学院`、`化工与化学学院`、`环境学院`、`机电工程学院`、`经管学院`、`生命科学和医学学部`、`实验报告`、`土木工程学院`、`未来技术学院`、`物理学院`、`薪火笔记社-竞赛`、`薪火笔记社-PPT及报告模板`、`询问中`、`仪器学院`。

### 课程页覆盖不足

`【公共课】` 远端已有大学物理、大学化学、电路类课程、工科数学分析、线性代数与空间解析几何、概率论与数理统计、近现代史纲要、马克思主义原理、毛概、习概、工程制图、理论力学等目录。本地只覆盖计算思维与人工智能、思想道德与法治两个课程页。

`交通科学与工程学院` 远端有交通管理、智能交通导论，本地只有学院索引。`电信学院`、`机电工程学院`、`物理学院`、`仪器学院` 等远端已观察到课程目录，但本地没有对应入口。

已观察到但本地缺失的首轮课程页候选包括：`电信学院/信号与系统`，`机电工程学院/数字电路`、`机电工程学院/电工技术`，`仪器学院/工程光学`，`物理学院/专业基础课` 下的 `光学`、`力学`、`原子物理`、`数学物理方法`、`热力学与统计物理`、`热学`、`理论力学`、`电动力学`、`电磁学`、`量子力学`，以及 `物理学院/细分专业课/光学`。

### 数学学院结构性失配

OpenList 中 `/数学学院` 已重组为 `/数学学院/专业基础课/...` 和 `/数学学院/细分专业课/...`。本地仍采用 `/数学学院/<课程>` 扁平结构，因此现有数学课程页的 `<OList path>` 大量失效。

当前已验证失效的数学学院页面共有 11 个：信息论基础讲义、初等数论、复变函数、实变函数、常微分方程、微分方程数值解院士完整版、数学分析、泛函分析、计算概论、集合论图论、高等代数 版本一。

### 重命名和孤儿页未建模

已发现的候选映射包括：`高等代数 版本一` 对应远端 `高等代数`，`微分方程数值解院士完整版` 对应远端 `微分方程数值解`，`【信息与计算科学】信息论基础讲义` 对应远端 `/数学学院/细分专业课/信息与计算科学/信息论基础`。

`集合论图论` 暂未在已抽样的 OpenList 数学学院新结构中找到对应目录，应列入孤儿页候选，由维护者决定删除、归档或重新映射。

### 运行时和维护流程不足

`OListItem.vue` 在 `fetchList(path, host)` 失败时缺少页面内错误状态。失效目录会让挂载阶段 promise rejected，用户无法获得清晰诊断。

仓库缺少自动化审计命令或 CI 检查来验证本地 `<OList path>` 是否仍存在于 OpenList。`README.md` 和 `CONTRIBUTING.md` 仍强调手写本地课程目录，但未说明如何参考 OpenList、如何迁移目录、如何处理例外。

## 方案对比

### 方案 A：最小止血

只修复当前 11 个数学学院失效 `<OList path>`，补一个基础错误提示。

优点是改动小、见效快。缺点是不能解决顶层分类和课程页覆盖不足，也不能防止后续继续漂移。

### 方案 B：OpenList 直接驱动站点

构建期拉取 OpenList manifest，自动生成课程树、课程页和 sidebar。

优点是和远端目录一致性最高。缺点是违反站点内 Markdown 必须为权威源的约束，也会让普通构建依赖远端网络，并削弱维护者对页面内容和结构的显式控制。

### 方案 C：Markdown 权威，OpenList 只读审计，手动全面迁移

保留手写 Markdown 作为站点权威源。新增只读审计工具读取 OpenList 和本地 Markdown，报告差距、失效路径和候选映射。维护者或 Agent 根据报告手动更新 Markdown。

优点是符合项目维护原则，普通构建不依赖远端网络，同时能系统性发现漂移并支持全面迁移。缺点是首次迁移仍需要人工或 Agent 手动处理大量页面。

## 选定方案

采用方案 C。

该方案把权威边界固定下来：Markdown 决定站点，OpenList 只提供外部资料目录事实。自动化负责“发现和阻止错误”，不负责“替维护者改页面”。这既能保留 VitePress 当前页面与 sidebar 生成方式，又能为全面迁移提供可重复的差距报告和验收标准。

## 详细设计

### 1. 权威源与迁移边界

站点内的权威源是手写 Markdown。VitePress 路由、sidebar、页面标题、课程说明和页面是否存在，都以仓库中的 Markdown 文件为准。

OpenList 不是站点结构生成源，也不允许脚本自动创建或更新课程 Markdown。OpenList 的角色是外部资料目录参考，用于检查每个 `<OList path>` 是否有效，以及提示哪些远端资源尚未被站点入口覆盖。

全面迁移的实际含义是：维护者或 Agent 根据审计报告手动调整 Markdown 目录结构，让站点入口尽量匹配 OpenList 当前目录。数学学院旧扁平页面应手动移动或改写到新层级。远端缺失的旧页面进入孤儿页报告，由维护者决定删除、保留说明或重新映射。

### 2. 只读审计工具

新增 `scripts/audit-openlist-structure.mjs`，并按需要新增 `scripts/openlist-audit.config.mjs` 保存例外配置。

配置文件包含 OpenList 连接参数和审计例外。默认连接参数为 `host: "https://olist-eo.jwyihao.top"`、`base: "Fireworks"`、`password: ""`，允许通过 CLI 参数或环境变量覆盖。仓库中不得提交真实密码或 secret。

审计工具职责：

1. 拉取 OpenList `/Fireworks` 目录树。深度语义以 `/Fireworks` 为 0 层，默认 `--depth=4`，以覆盖 `/数学学院/细分专业课/信息与计算科学/信息论基础` 这类结构；支持通过参数调整深度，并设置请求超时、并发上限、失败重试上限和最大目录数，避免无界递归或 CI 中请求过量。
2. 扫描本地 Markdown 课程页，提取页面路径、标题、`<OList path="..." />`、`<OListItem path="..." />` 和无显式 path 的 `<OList />` / `<OListItem />`。无显式 path 必须规范化为 `/`，只校验远端根目录可访问，不应算作某个课程目录已被本地 Markdown 覆盖。
3. 规范化本地和远端路径，计算远端有但本地无、本地有但远端无、当前失效 `<OList path>`、旧页面候选映射和孤儿页候选。
4. 默认只输出控制台报告；显式传入 `--report docs/superpowers/reports/openlist-audit.md` 时，才可写入非课程文档位置。报告文件是审计产物，不是课程页面生成物。

Markdown 扫描范围应镜像当前 VitePress 内容页边界，排除 `docs/**`、`parts/**`、`README.md`、`CONTRIBUTING.md`、`team.md` 和报告目录，避免把设计文档、贡献指南或片段示例当作课程页。提取组件时必须忽略 fenced code block 和 inline code。首版只支持静态 `path` 的单引号或双引号、以及裸组件；动态 `:path` / `v-bind:path` 或非默认 `host` 等参数应报告为 `warning` 或配置错误，不应猜测路径。

审计报告只用于人工决策，不进入 sidebar，不替代任何课程 Markdown 页面。

数据流为：OpenList API 与本地 Markdown 扫描结果进入规范化路径树，再进入 diff/report，最后由维护者或 Agent 手动编辑 Markdown 并重新运行审计验证。

普通 VitePress 构建不调用该审计工具。CI 可以单独运行审计，把它作为维护质量检查。

### 3. 手写 Markdown 迁移规则

审计报告中未被例外忽略的 OpenList 顶层目录，应由维护者或 Agent 手动确认后补本地 `index.md`，这样 sidebar 和站内搜索可以发现学院或分类入口。当前 18 个 OpenList 顶层目录是首轮候选清单，后续以审计报告为准。

OpenList 中承载课程资料的目录应作为课程页候选进入报告。是否创建课程 Markdown、页面标题和说明内容，仍由维护者或 Agent 手动编辑决定。课程页至少包含课程标题、简短说明、资料下载小节和 `<OList path="远端目录" />`。

纯中间分组目录也应进入补页候选，例如 `/数学学院/专业基础课`、`/数学学院/细分专业课`、`/物理学院/专业基础课`。维护者或 Agent 手动确认后补 `index.md`，否则 sidebar 无法表达 OpenList 的真实层级变化。

临时目录或不适合公开成课程入口的目录，例如 `询问中`，可以保留为分类页或加入审计例外。例外必须显式配置，避免每次报告都产生重复噪音。

非普通课程顶层目录不得默认按“学院/课程”结构迁移。`实验报告`、`薪火笔记社-竞赛`、`薪火笔记社-PPT及报告模板` 应优先作为资料分类页处理：可以建立顶层 `index.md` 作为站点入口，并在页面中说明其不是普通课程分类；其下级目录只有在确认为具体课程或稳定专题时才建立课程页。`询问中` 属于临时或待确认目录，默认不建立普通课程页，可建立说明型分类页或加入审计例外。所有例外必须写明原因。

数学学院首轮迁移前必须维护一张手动映射表，至少覆盖当前 11 个失效页：旧 Markdown 路径、旧 `<OList path>`、目标 OpenList 路径、目标 Markdown 路径、处理状态。已确认映射应直接列入表中，例如 `高等代数 版本一` -> `/数学学院/专业基础课/高等代数`，`微分方程数值解院士完整版` -> `/数学学院/专业基础课/微分方程数值解`，`【信息与计算科学】信息论基础讲义` -> `/数学学院/细分专业课/信息与计算科学/信息论基础`。暂未确认的条目不得静默迁移，应标记为待确认或孤儿页候选。

首轮手动迁移顺序：

1. 修复数学学院 11 个失效页，并把数学学院重排到 `专业基础课` 和 `细分专业课` 层级。
2. 补齐 OpenList 已有但本地缺失的顶层目录页。
3. 为公共课、交通科学与工程学院、电信学院、机电工程学院、物理学院、仪器学院等已观察到的课程目录补课程页。
4. 根据审计报告继续处理 warning 和 info，逐步收敛未覆盖目录与孤儿页。

### 4. `OListItem.vue` 错误处理

`OListItem.vue` 应为初始加载和懒加载分别增加错误状态。

初始加载失败时，页面正文仍应可读，资料下载区域显示明确失败提示。提示至少包含当前 `path`，说明资料目录加载失败，并建议刷新或联系维护者。

懒加载子目录失败时，应在对应节点下显示失败状态或通过 toast 提示，不能静默失败。失败后应保留可重试状态，不能永久显示 loading skeleton；用户应能重新展开或刷新页面重试。

错误处理不改变下载源选择、校园网检测和 OpenList API 形态。本轮只保证失效路径或网络异常不会让页面体验退化为无提示失败。

### 5. 构建期 OpenList 依赖处理

现有 `.vitepress/theme/components/alist.data.mts` 是 VitePress data loader，会在构建期调用 `fetchList()` 拉取 OpenList 根目录。该行为与“普通 `bun run docs:build` 不依赖远端 OpenList 网络”的目标冲突，必须在本轮处理。

允许的处理方式包括：移除构建期远端拉取、让 data loader 返回静态空兜底数据、或捕获网络错误并返回空数组。根目录 `<OList />` 的真实资料列表应在客户端运行时加载，并通过 `OListItem.vue` 的错误状态处理失败。构建期不得因为 OpenList 网络不可用、API 异常或远端超时而失败。

### 6. 审计结果分级与 CI

审计结果分为三类：

1. `error`：本地 `<OList path>` 指向 OpenList 不存在目录。当前数学学院 11 个失效路径属于这一类。CI 应失败。
2. `warning`：OpenList 有目录但本地无 Markdown 入口，表示资源不易被 sidebar 或站内搜索发现。全面迁移阶段应逐步清零，但初期可以只报告不失败。
3. `info`：重命名候选、孤儿页候选、例外目录命中等，需要人工判断。

CI 初期只阻止新增或残留 `error`。`warning` 和 `info` 输出到报告，供后续人工迁移。等 Markdown 课程树基本完成迁移后，再考虑把部分 `warning` 提升为失败。

审计命令退出码：`0` 表示审计完成且无 `error`；`1` 表示审计完成但发现 `error`，例如本地 `<OList path>` 指向 OpenList 中明确不存在的目录；`2` 表示审计未能完成，例如网络超时、DNS 失败、OpenList API 不可用、响应非 JSON 或协议异常；`3` 表示本地参数、配置或输入文件错误。

只有 OpenList 返回 `object not found` 语义时，才把目标路径判定为不存在目录。DNS、超时、非 JSON、HTTP 5xx、鉴权异常等不得归为课程路径 `error`，应归为网络、API 或配置失败。

CI 接入顺序是硬约束：先交付只读审计工具和控制台报告，不开启 fail gate；再修复 `OListItem.vue` 错误状态并手动迁移数学学院，确认当前 11 个已知 `error` 清零；最后在独立 audit job 或独立 workflow 中启用 fail-on-error。普通部署 build job 不应因为 OpenList 临时不可用而失败。

例外配置只声明忽略哪些远端目录缺页警告、哪些本地页面允许无 OList、哪些旧页暂时保留。例外配置不声明自动改写 Markdown 的规则。默认不允许例外压制无效 `<OList path>` 的 `error`；若确需临时压制，必须显式写明原因、责任人或清理条件，并在报告中作为 `info` 暴露。

`warning` 初期不使 CI 失败，但报告必须稳定列出 warning 总数和详情。迁移基本完成后，可以逐步设置 warning budget 或禁止新增 warning。

### 7. 文档更新

`README.md` 应更新项目结构说明，明确 Markdown 是站点入口权威，OpenList 是资料目录参考。目录示例应避免继续暗示当前少量本地目录就是完整课程树。

`CONTRIBUTING.md` 应增加课程页维护流程：新增、移动或删除课程页时，维护者需要确认对应 OpenList 目录路径，并运行审计命令检查 `<OList path>` 是否存在。

`CONTRIBUTING.md` 的课程页维护流程应包含检查清单：确认目标 Markdown 路径、确认页面标题和说明、确认 `<OList path>` 指向真实 OpenList 目录、判断页面类型是课程页/中间分组/非普通资料分类、运行审计命令、检查审计报告中是否新增 `error`。不得把审计报告当作站点入口提交到课程目录。

文档还应说明远端目录新增后的处理方式：先看审计报告，再由维护者或 Agent 手动添加 Markdown 分类页或课程页；脚本不会自动生成课程页面。

## 验证计划

本地验证：

1. 执行 `bun scripts/audit-openlist-structure.mjs`，确认能输出 error、warning、info 数量和详细路径。
2. 执行无依赖脚本自检，例如 `bun scripts/audit-openlist-structure.mjs --self-test`，覆盖路径规范化、Markdown `<OList>` 提取和 diff 逻辑。自检至少覆盖中文路径、重复斜杠、无显式 path 的 `<OList />`、多个 `<OList>`、代码块中的 `<OList>` 不被识别、不存在远端目录。
3. 手动迁移数学学院后再次执行审计，确认当前 11 个已知 error 被清除或转为明确孤儿页处理。
4. 执行 `bun run docs:build`，确认 VitePress 构建通过，sidebar 能反映手写 Markdown 新结构。
5. 在 OpenList 不可用或模拟请求失败的情况下执行 `bun run docs:build`，确认普通构建不因 `.vitepress/theme/components/alist.data.mts` 的远端加载失败而失败。
6. 在本地打开包含有效和无效测试路径的页面，确认 `OListItem.vue` 错误状态可见且不影响页面正文。

CI 验证：

1. 在数学学院已知 `error` 清零后，增加独立审计 job 或独立 workflow，只在审计命令返回路径 `error` 时失败。
2. 确认普通构建步骤不依赖 OpenList 网络可用性。
3. 审计访问 OpenList 失败时应返回明确错误和非零退出码，且退出码能区分远端不可用、路径 `error`、本地配置错误。

文档验证：

1. 检查 `README.md` 不再把当前少量本地目录描述为完整课程树。
2. 检查 `CONTRIBUTING.md` 明确说明 Markdown 权威源、OpenList 参考源、审计命令和手动迁移流程。
3. 检查审计报告不会写入课程目录，不会被误认为自动生成课程页。

## 风险与回滚

审计工具依赖 OpenList 网络和 API 稳定性。如果远端临时不可用，CI 审计可能失败。缓解方式是普通构建不依赖审计，并在 CI 中把网络失败信息与路径失效信息区分清楚。

全面手动迁移会产生大量 Markdown 目录改动，可能让 sidebar 结构短期变化明显。本轮不考虑旧 URL 迁移，因此旧路径访问失效是已接受约束。

例外配置可能被滥用，导致长期 warning 被隐藏。配置中每个例外都应有简短原因，后续审查时可以判断是否仍需要保留。

`OListItem.vue` 错误状态如果设计过重，可能影响正常页面视觉。应保持提示简洁，只在资料列表区域内呈现，不改变页面主体结构。

如果审计工具或 CI 接入造成维护阻塞，可以先回滚 CI 强制失败步骤，保留本地审计命令和报告能力。课程 Markdown 手动迁移本身不依赖 CI。

## 接受标准

1. 存在只读 OpenList 审计命令，能扫描远端 `/Fireworks` 与本地 Markdown，并输出 error、warning、info。
2. 审计命令不会自动创建、移动、删除或修改课程 Markdown 页面。
3. 当前本地 `<OList path>` 指向不存在远端目录时，审计结果标记为 `error` 并能使 CI 失败。
4. `OListItem.vue` 在初始加载和懒加载失败时显示清晰错误状态，页面正文仍可阅读。
5. 数学学院失效页面有明确的手动迁移方案，`集合论图论` 等远端未匹配页面进入孤儿页报告。
6. 缺失顶层分类和课程页能在审计报告中稳定列出，供维护者逐步手动补齐。
7. `README.md` 和 `CONTRIBUTING.md` 说明 Markdown 是站点权威源，OpenList 是资料目录参考，并写明审计与手动迁移流程。
8. `.vitepress/theme/components/alist.data.mts` 不再让普通 VitePress 构建依赖 OpenList 成功响应。
9. `bun run docs:build` 在没有访问 OpenList 的情况下仍能完成站点构建。
